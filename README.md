## Тестовое задание на позицию Fullstack разработчика (Python + React)

**Вводные:**
1. Здесь представлен MVP проект файлообменника. Он позволяет загружать файлы, проверяет их на подозрительный контент и отправляет алерты;
2. Репозиторий содержит в себе бэкенд и фронтенд части;
3. В обоих частях присутствуют баги, неоптимизированный код, неудачные архитектурные решения.

**Задачи:**
1. Проведите рефакторинг бэкенда, не ломая бизнес-логики: предложите свое видение архитектуры и реализуйте его;
2. Пофиксите найденные баги и оптимизируйте код, напишите тесты;
3. Сделайте рефакторинг фронтенда, разбейте логику на слои;
4. Добавьте пагинацию (бэк + фронт) на эндпоинты GET /files и GET /alerts.

**Запуск:**
1. ```docker compose -f docker-compose.dev.yml up```
2. ```docker exec -it backend alembic upgrade head```


**Открыть фронт:** ```http://localhost:3000/test``` 

**Открыть бэк:** ```http://localhost:8000/docs```

## В ходе выполнения задания реализовано:
1. Рефакторинг бэкенда (Clean Architecture (Layered))
2. Рефакторинг фронтэнда (Feature-sliced design)
3. Добавлена пагинация в API и виджет на фронт. Для удобства тестирования фронт начинает пагинировать более двух записей.

**Бэкенд:**
Слои: API => USE_CASES => REPOSITORIES
Код разделен на Domain и Infrastructure разделы, где domain - схемы приложения, юз-кейсы, бизнес-логика, а infrastructure - детали конкретной реализации - схемы БД, репозитории, Pydantic модели, реализация используемых технологий (celery, storage).
Инжект зависимостей вынесен в dependencies.py с помощью FastAPI's Depends() (что в свою очередь в опр. момент вызвало цикличный импорт т.к. зависимости импортируют друг друга и пронизывают всё приложение).
Способ передачи данных ВНУТРИ приложения и СНАРУЖИ разделены в виде DTO \ RESPONSE схем соответственно - таким образом, мы контролируем, что именно видит клиент и уверены в сокрытии важных данных от посторонних.
Для реализации DRY в виде "единого источника правды" схемы (и енумы) с бэкенда передаются на фронт с помощью openapi-typescript - это должно происходить при коммите в процессе ci\cd.
Добавлен Pydantic-settings для более чистого контроля над настройками и окружением проекта.
**Фронтэнд:**
Слои: app => pages => widgets => features => entities + shared
Реализован классический вид Feature Sliced Design, упрощающий некоторые структуры файлов (но все ещё громоздкий).
Основная логика содержится в pages/Dashboard, которая в свою очередь импортирует entities/*/ui как способы отображения сущностей (они не считаются виджетами т.к. недостаточно умные и представляют собой простую репрезентацию объекта). Управление пагинацией и запросом данных вынесено в хуки entities/*/useEntity.tsx.
Features - "умные" компоненты проекта, которые инкапсулируют в себе определенную бизнес логику. В данном случае выделена только одна фича - кнопка загрузки файла, которая вызывает модальное окно и валидирует данные.
**Тесты:**
Тесты поделены на два вида - integration, unit.
Интеграционные тесты проверяют работу компонентов друг с другом, они более тяжелые и иногда используют реальную реализацию (в зависимости от предмета теста). На текущий момент в них входят тесты важных API, которые проверяют появление артефактов в БД и вызов задач, и Persistence тесты, которые проверяют, что репозиторий сохраняет данные в том же виде, в котором они приходят.
Юнит тесты никогда не используют детали реализации и проверяют только чистые функции бизнес-логики - они выполняются быстро и должны быть использованы разработчиком перед коммитом.

**Known issues**
Зависимости в Celery захардкожены, т.к. это единственный найденный мною способ передать зависимость в задачу селери при текущей реализации.
Нет тестов celery задач, чтобы знать когда именно хардкод зависимостей в задачах отвалится (т.к. Celery не дружит с DI приложения)
Нужно добавить Storage класс, который будет работать со стримом чанков, а не с полным-файлом-в-памяти. (сейчас для защиты от перегрузки файлами добавлено ограничение на размер файла в 15 мегабайт) 
Нет тестов немалой части API эндпоинтов.
